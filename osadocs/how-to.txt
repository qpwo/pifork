# Write a file safely with open for access; close in the error branch, then close after success.
osascript <<'AS'
set f to POSIX file "/tmp/osa-write-demo.txt"
set h to open for access f with write permission
try
    set eof of h to 0
    write ("hello" & linefeed) to h
on error e number n
    close access h
    error e number n
end try
close access h
AS

osascript -e 'set alpha to {1, 2, {"a", "b"}}' -e 'copy alpha to beta' -e 'set item 2 of item 3 of alpha to "change"' -e 'set item 1 of beta to 42' -e 'return {alpha, beta}'

# Footnotes
# ------------------------------------------------------------------------------

# [1] `osascript -s flags`: The -s flag modifies output style. The four main flags are:
#     -s h: Human-readable (default). Easy to read, but can be ambiguous. `{"a", "b"}` and `{"a, b"}` both print as `a, b`.
#     -s s: Recompilable source. Unambiguous, machine-readable output. `{"a", "b"}` prints as `{"a", "b"}`.
#     -s o: Routes normal output and error messages to stdout. The default is to send errors to stderr.
#     -s e: Routes normal output to stderr.

# [2] `osacompile -x` (Execute-Only): This flag creates a script that can be run, but whose source code cannot be viewed or
#     edited in Script Editor. This is a form of obfuscation, useful for distributing scripts without revealing the
#     implementation. It is not a robust security feature, as the code can still be recovered by determined users.

# [3] `osacompile -s` (Stay-Open): When compiling an applet (.app), the -s flag makes it a "stay-open" application.
#     A normal applet launches, runs its `on run` handler, and then quits. A stay-open applet remains running after
#     the `run` handler finishes. If it has an `on idle` handler, that handler will be called periodically, making it
#     suitable for background tasks. The `idle` handler can `return <seconds>` to set the delay until its next call.

# [4] `alias` vs `file` vs `POSIX file`: These specify filesystem paths with important differences.
#     - `alias`: A dynamic reference to an item that *must exist* when the alias is created. The alias tracks the item by
#       its unique filesystem ID, so it will not break if the item is moved or renamed.
#     - `file`: A static reference using a colon-separated HFS-style path (e.g., "Macintosh HD:Users:me:Desktop:file.txt").
#       The object does not need to exist when the reference is created.
#     - `POSIX file`: A static reference using a slash-separated POSIX path (e.g., "/Users/me/Desktop/file.txt"). The object
#       also does not need to exist. This form is most compatible with `do shell script` and Unix tools. Note that `~` is
#       not expanded; use `(path to home folder)` to get the home directory path.

# [5] `my` and `of me` in `tell` blocks: Inside a `tell application "..."` block, every command is sent to the target
#     application by default. If your script defines its own handlers, you must use `my handlerName()` or `handlerName() of me`
#     to specify that the handler is part of your script, preventing AppleScript from trying to send the command to the
#     target application (which would cause an error).

# [6] `set` vs `copy`: For mutable objects (lists, records, dates), these commands have different behaviors.
#     - `set b to a`: `b` becomes another reference to the exact same object in memory as `a`. Modifying `a` will be reflected
#       in `b`, and vice versa. It's like creating a pointer.
#     - `copy a to b`: `b` becomes a new, completely independent, "deep" copy of `a`. Modifying `a` after the copy will have
#       no effect on `b`. For immutable values like numbers and strings, the distinction is academic.

# [7] `a reference to`: For very large lists, repeatedly accessing items or appending to the end can be slow, as AppleScript
#     may re-process the list on each operation. Creating `set listRef to a reference to bigList` provides a more direct handle.
#     Subsequent operations on `listRef` (e.g., `item 10000 of listRef` or `copy n to the end of listRef`) are significantly
#     faster.

# [8] `property`: A `property` is like a global variable, but its value persists across runs of a compiled script. The value is
#     stored within the script file itself and is only reset when the script is recompiled. This is useful for maintaining state,
#     such as a counter or user preference, without needing an external file.

# [9] `do shell script`: This command uses a non-interactive `/bin/sh` shell. It does not load your personal `.zshrc` or
#     `.bash_profile`, so your PATH and custom aliases/functions will not be available. Use full paths to executables. Use
#     `quoted form of posixPath` to safely pass file paths or other user text to the shell, which prevents errors and security
#     issues from spaces and special shell characters.

# [10] Interleaved Handlers: AppleScript supports handlers with "interleaved" or "keyword-style" arguments, similar to
#      Objective-C. A handler like `on areaOfRectangleWithWidth:w height:h` is called with `my areaOfRectangleWithWidth:5 height:10`.
#      The actual name of the handler is `areaOfRectangleWithWidth:height:`. These calls must be explicitly targeted with `my`,
#      `me`, or another object reference; a bare call is a syntax error.
